# レビュー体制仕様

## 目的

この文書は、`danmaku` におけるレビュー体制を定義するための正本である。
レビュー担当者をサブエージェントとして使う前提で、役割、入力、出力、運用順、最終判断の流れを固定する。

## 基本方針

- レビュー担当者は `3 人` とする
- 各レビュー担当者は、それぞれ異なる観点を持つ
- 1 人で全てを判断しない
- サブエージェントは `レビュー専任` とし、実装の正本更新は主担当エージェントが行う
- サブエージェントの出力は `助言` であり、最終採用判断は主担当エージェントが行う
- ユーザーが単に `レビューして` と依頼した場合も、この文書の運用を標準とする

## 役割分担

### 1. レビュー担当者A

- 技術設計とコード品質を主に見る
- 対象は、責務分離、構成、命名、保守性、Elixir らしさ、実装リスク
- 前提として主に読む文書は次とする
- [ARCHITECTURE.md](ARCHITECTURE.md)
- [HARNESS_SPEC.md](HARNESS_SPEC.md)
- [ELIXIR_CODING_RULES.md](ELIXIR_CODING_RULES.md)
- [ELIXIR_CREATOR_NOTES.md](ELIXIR_CREATOR_NOTES.md)
- [LIBRARY_VERSIONS.md](LIBRARY_VERSIONS.md)

### 2. レビュー担当者B

- ゲーム体験とゲーム仕様を主に見る
- 対象は、遊びやすさ、理不尽さ、視認性、弾幕の読みやすさ、UI の伝わりやすさ
- 前提として主に読む文書は次とする
- [GAME_SPEC.md](GAME_SPEC.md)
- [DESIGN_SPEC.md](DESIGN_SPEC.md)
- [GAME_REVIEW_RULES.md](GAME_REVIEW_RULES.md)

### 3. レビュー担当者C

- 整理、工程、文書整合性を主に見る
- 対象は、仕様の抜け漏れ、文書の役割衝突、工程との整合、タスク分割の妥当性
- 前提として主に読む文書は次とする
- [PLAN.md](PLAN.md)
- [ROADMAP.md](ROADMAP.md)
- [WATERFALL_PLAN.md](WATERFALL_PLAN.md)
- [STATUS_BOARD.md](STATUS_BOARD.md)
- [RISK_NOTES.md](RISK_NOTES.md)
- [TECH_SECRETARY_PROFILE.md](TECH_SECRETARY_PROFILE.md)
- 必要に応じて、変更対象の正本文書も読む

## 主担当エージェントの役割

- レビュー依頼の対象を決める
- 各レビュー担当者へ必要な文脈だけ渡す
- 出力を比較し、矛盾や重複を整理する
- 採用する指摘と保留する指摘を決める
- 必要なコード、文書、タスク管理へ反映する

## サブエージェントに渡す入力

各レビュー依頼では、少なくとも次を渡す。

- 対象ファイル
- レビュー目的
- 関連する仕様書
- 変更前提
- どの観点で見てほしいか

必要に応じて渡すもの:

- 差分
- 現在の工程
- 関連する設計方針
- 既知の懸念点

## サブエージェントに期待する出力

各レビュー担当者は、次の形式で返す。

- 指摘事項
- 指摘の重要度
- なぜ問題か
- どう直すとよいか
- 該当ファイル
- 該当箇所

レビュー結果は、できるだけ次の区分で整理する。

- `必須修正`
- `修正推奨`
- `確認事項`
- `問題なし`

## レビュー実行の順序

### 1. 依頼対象を決める

- コード変更か
- 仕様変更か
- 文書変更か
- デザイン変更か

### 2. 必要なレビュー担当者を選ぶ

- 技術寄りなら A を優先する
- ゲーム体験寄りなら B を優先する
- 文書や工程寄りなら C を優先する
- 大きい変更では 3 人全員に依頼してよい

### 3. 各サブエージェントへ個別に依頼する

- 全員に同じ投げ方をせず、観点を明示する
- 不要な文脈は渡しすぎない
- 役割外の判断は主担当へ戻す

### 4. 主担当エージェントが統合する

- 重複指摘をまとめる
- 矛盾する指摘を比較する
- 採用、不採用、保留を整理する

### 5. 必要な反映を行う

- コード修正
- 文書修正
- 仕様確認
- タスク更新

## レビュー対象ごとの使い分け

### コード変更

- A は原則必須
- B はゲーム体験へ影響する場合に実施する
- C は仕様や文書整合性へ影響する場合に実施する

### ゲーム仕様変更

- B を原則必須とする
- C で文書整合性を確認する
- A は実装制約が大きい場合に実施する

### 設計変更

- A を原則必須とする
- C で工程と文書整合性を確認する
- B はゲーム体験へ波及する場合に実施する

### 文書変更

- C を原則必須とする
- A と B は必要時のみ呼ぶ

## 指摘が衝突した場合

- 主担当エージェントが衝突を整理する
- 技術妥当性は A を重く見る
- ゲーム体験は B を重く見る
- 文書整合性と工程整合は C を重く見る
- 最終判断に迷う場合は、主担当エージェントがユーザーへ確認する

## レビューの記録

レビュー結果は、必要に応じて次へ反映する。

- 仕様変更なら該当仕様書
- 工程変更なら [STATUS_BOARD.md](STATUS_BOARD.md)
- リスク追加なら [RISK_NOTES.md](RISK_NOTES.md)
- コーディング判断なら [ELIXIR_CODING_RULES.md](ELIXIR_CODING_RULES.md)

レビュー結果そのものを毎回長文で恒久保存することは前提にしない。
恒久保存が必要なのは、採用された判断だけとする。

## この体制で避けること

- 3 人全員に毎回同じ依頼を投げる
- 観点を指定せずに雑にレビューさせる
- サブエージェントに最終判断まで委ねる
- 指摘を集めるだけで整理しない
- レビュー結果を反映せず放置する

## 今後決めること

- 3 人の正式名称
- 3 人それぞれの詳細プロンプト
- レビュー依頼テンプレート
- レビュー結果の要約テンプレート

## 一言でまとめると

このレビュー体制は、`3 人の専門レビュー担当者に見てもらい、主担当エージェントが統合して最終判断する仕組み` である。
