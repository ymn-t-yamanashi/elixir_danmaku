# Elixir Web弾幕シューティング仕様書

## 目的

この文書は、`danmaku` のゲーム内容と挙動を定義する。
ゲームロジックは Elixir 側で扱い、表示は 2.5D の Babylon.js 描画で表現する。

## 本文

この仕様書は、Elixir主導で開発するWeb弾幕シューティングのゲーム仕様を定義する。
ゲーム性は2D、見た目は3D表示の `2.5D` とし、東方Project風の読みやすく美しい弾幕戦を目指す。

関連文書:

- [PLAN.md](../project/PLAN.md)
- [ARCHITECTURE.md](ARCHITECTURE.md)

## ゲーム概要

- ジャンル: 縦スクロール弾幕シューティング
- プレイ人数: 1人
- 表示: 3D演出付き `2.5D`
- ゲームロジック: Elixir側で管理
- 描画: `Babylon.js`

## コアコンセプト

- 自機は小さく、細かい回避が成立する
- 弾幕は高密度でも読みやすさを保つ
- ランダム性より規則性と学習性を重視する
- ボス戦とフェーズ進行をゲーム体験の中心に置く
- 3D演出は迫力のために使い、判定のわかりやすさは崩さない

## 画面仕様

### カメラ

- 基本は固定の俯瞰寄りカメラ
- プレイ中は視認性優先で大きく動かさない
- ボス登場やフェーズ切替時のみ軽い演出変化を許容する

### 背景

- 奥行きのある3D背景を表示する
- 和風または幻想的な建築、霧、光、浮遊物などで世界観を演出する
- 背景は派手すぎず、弾と敵の視認性を妨げない

### プレイ領域

- ゲームルール上は2D平面
- 見た目では3D空間上に配置する
- Elixir座標の `x, y` を描画側で `x, z` に対応させる

## UI仕様

### 常時表示

- スコア
- 残機
- ボム数
- ボスHPバー
- 現在フェーズ名

### 表示方針

- UIは画面端に整理して配置する
- プレイ領域を圧迫しない
- ボス戦時の情報は視線移動が少ない位置に置く

## プレイヤー仕様

### 基本

- 自機は画面下部を中心に移動する
- 当たり判定は見た目より小さい
- 高速移動と低速移動を持つ

### 入力

- 移動: `WASD` または矢印キー
- ショット: 1ボタン
- ボム: 1ボタン
- 低速移動: 1ボタン

### 挙動

- 入力はクライアントから送信し、移動計算はElixir側で行う
- 低速移動中は精密回避しやすい速度にする
- 画面外へは出られない

## ショット仕様

- ショットは自機前方へ発射される
- 初期はシンプルな直線ショットから始める
- 将来的に強化段階やオプション追加を検討できる設計にする

## ボム仕様

- 緊急回避用のリソースとする
- 発動中は広範囲の敵弾除去または無効化を行う
- 演出は派手でも、判定のわかりやすさを優先する

## 敵仕様

### 雑魚敵

- 道中で出現する
- シンプルな移動と単純な弾幕を担当する
- スコア源として機能する

### ボス

- ゲーム体験の中心
- 複数フェーズを持つ
- フェーズごとに明確な弾幕テーマを持つ
- HPバーとフェーズ名を表示する

## 弾仕様

### 基本方針

- 弾は見た目の美しさと避けやすさの両立を目指す
- 高密度でも自機周辺の認識がしやすいことを優先する
- 当たり判定は2Dロジックを正とする

### 初期パターン

- 直線弾
- 扇状弾
- 円形拡散弾
- 渦巻き弾
- 時間差発射弾

### 設計方針

- 弾幕は規則を持ち、観察と学習で攻略可能にする
- 無秩序なランダム弾は避ける
- ボスごとに象徴的なパターンを持たせる

## 当たり判定仕様

- ゲーム内の正しい判定はすべてElixir側で処理する
- 自機判定は小さな円を基本にする
- 敵弾判定も単純な形状を基本にする
- 見た目の3D形状は判定に使わない

## ゲーム進行

### 基本ループ

1. プレイヤー入力を受け取る
2. Elixir側で自機、敵、弾の状態を更新する
3. 当たり判定とスコア更新を行う
4. 更新済み状態をクライアントへ送信する
5. クライアントがBabylon.jsで描画する

### ステージ進行

- 道中
- 中ボス
- 道中後半
- ボス戦
- リザルトまたは次ステージ

## 演出仕様

### 3D演出

- 背景の奥行き
- ライティング
- 薄い霧や空間エフェクト
- ボス登場演出
- フェーズ切替演出

### 演出の制約

- カメラを揺らしすぎない
- 弾の視認性を下げない
- 被弾時に状況把握できなくなる演出を避ける

## 初期実装範囲

### 必須

- 自機1体
- 敵1種類
- 自機ショット
- 敵弾
- 当たり判定
- 残機
- ゲームオーバー
- スコア
- ボス1体
- ボス2フェーズ以上

### 後回し

- 難易度選択
- リプレイ
- 永続的なランキング保存
- 複雑な成長要素
- 大量の3Dモデル作成

## 保存方針

- DBは使わない
- ゲーム状態は実行中メモリのみで保持する
- スコアはプレイ中とリザルト画面で表示する
- アプリ再起動後も残る保存機能は初期仕様に含めない

## 実装判断の原則

- 迷ったら視認性を優先する
- 迷ったらゲームロジックはElixir側へ寄せる
- 迷ったら演出より操作感を優先する
- 迷ったら複雑な抽象化より小さい構成を選ぶ
