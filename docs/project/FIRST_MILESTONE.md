# 最初のマイルストーン

## 目的

この文書は、`danmaku` が最初に到達すべき実装到達点を示す。
初期実装の範囲を明確にして、着手判断を揃える。

## 本文

この文書は、`danmaku` の最初の実装単位を明確にするための文書である。
最終目標は弾幕シューティングだが、最初のマイルストーンではそこまで求めない。
目的は「このプロジェクトが実際に前へ進む」最小の成功状態を定義することにある。

## この段階でやらないこと

- ボス戦
- スペルカード演出
- 高密度弾幕
- 音の作り込み
- 複雑なUI演出
- 最終品質の背景表現
- 追加コンテナ構成

## 最初のマイルストーンで達成すること

次の 5 点を達成したら、このマイルストーンは完了とする。

1. Docker 前提で Phoenix アプリが起動する
2. 3Dシーン上に自機が1体表示される
3. 入力に応じて自機が移動する
4. 敵が1体表示される
5. 敵が単純な弾を1種類だけ発射する

## マイルストーンの成功条件

### 必須条件

- ブラウザで画面が開く
- 自機が下部中央に表示される
- `WASD` または矢印キーで移動できる
- 自機が画面外に出ない
- 敵が上部に1体表示される
- 敵弾が一定間隔で発射される
- 弾が3D表示される
- プレイが極端に重くならない

### まだ不要なもの

- 被弾判定
- ゲームオーバー
- スコア
- ボム
- ボスHPバー
- スペルカード名
- 複雑な背景演出

## この段階の設計方針

- ゲームルールは2Dで扱う
- 見た目だけ3Dにする
- 自機と敵と弾だけに対象を絞る
- Babylon.js は最小のシーン構築に留める
- Elixir 側は位置更新と単純な弾生成だけを担う

## 最初の表示内容

### 自機

- 画面下部中央に1体
- 簡素なメッシュまたは板ポリでよい
- 足元のリング演出はあってもよいが必須ではない

### 敵

- 画面上部中央に1体
- 固定位置でもよい
- まだ複雑な移動は不要

### 弾

- 1種類だけ
- 直線で下方向へ進む
- 視認しやすい発光表現をつける

### 背景

- 簡易な3D空間でよい
- 最初は暗めの空間と奥行きだけでもよい
- 和風建築の作り込みはこの段階では不要

## 入力仕様

- 移動入力だけ実装する
- ショット入力はこの段階では不要
- 低速移動もこの段階では不要

## 通信仕様

- クライアントは移動入力を送信する
- Elixir 側は自機位置を更新する
- 敵弾の生成も Elixir 側で行う
- クライアントは受け取った状態を表示する

## 技術的な確認ポイント

- Docker で Phoenix が正常起動するか
- Babylon.js の初期表示が安定するか
- Phoenix Channels 経由で状態を送受信できるか
- Elixir 側の更新ループを無理なく組めるか

## この段階で確認したいリスク

- Elixir 主導更新の最小構成が重くないか
- 2Dロジック + 3D表示の役割分担が無理なく機能するか
- 画面上で自機と弾の視認性が確保できるか

## 完了後に次へ進めるもの

このマイルストーン完了後に、次の要素へ進める。

- 自機ショット
- 被弾判定
- ゲームオーバー
- スコア
- より美しい弾幕パターン

## 結論

最初のマイルストーンは、「弾幕シューティング完成」ではない。
「Docker 上で Phoenix と Babylon.js を使い、自機、敵、単純な敵弾が3D表示で動くこと」を成功条件とする。
これを達成できれば、`danmaku` は企画段階から実装段階へ安全に移行できる。
