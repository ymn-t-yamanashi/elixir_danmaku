# プロジェクト危険ポイント整理

## 目的

この文書は、`danmaku` で破綻しやすい点と注意事項を整理する。
実装前にリスクを確認して、判断の迷いを減らすための正本である。

## 本文

この文書は、このプロジェクトが破綻しやすい点を事前に整理するためのメモである。
目的は不安を増やすことではなく、危険箇所を早い段階で見える化し、進め方を安定させることにある。

## 結論

このプロジェクトは無謀ではない。
ただし、難しい箇所を同時に抱えやすいため、順番を間違えると一気に破綻しやすい。

特に注意すべきなのは次の 5 点である。

1. Elixir主導リアルタイム更新の負荷
2. 3D表示による視認性低下
3. 仕様の過剰拡張
4. 演出優先で基礎実装が遅れること
5. 開発環境の複雑化

## 危険ポイント 1: Elixir主導リアルタイム更新

### 危険内容

- 自機、敵、弾、当たり判定を Elixir 側で毎フレーム更新するため、弾数が増えると負荷が上がる
- 状態同期の設計が雑だと、通信量が増えてプレイ感が悪化する

### 破綻の兆候

- 弾数を増やした瞬間に更新遅延が目立つ
- 入力に対して自機の反応が鈍い
- クライアントへの送信データが肥大化する

### 避け方

- 最初は弾数を少なくする
- まず 1 パターンだけで負荷を見る
- 送信データは必要最小限にする
- 早い段階で簡単な性能測定を入れる

## 危険ポイント 2: 3D表示による視認性低下

### 危険内容

- 背景、光、カメラ、エフェクトを盛りすぎると、弾や自機が見えにくくなる
- 東方風の見た目を狙うほど、演出過多になりやすい

### 破綻の兆候

- 美しいが遊びにくい
- 弾の当たり判定が読みにくい
- 自機位置を見失いやすい

### 避け方

- 判定は2D、見た目だけ3Dという原則を崩さない
- 背景より弾と自機の可読性を優先する
- 通常プレイ中のカメラ演出を控える
- まずは簡素な見た目で遊べるかを確認する

## 危険ポイント 3: 仕様の過剰拡張

### 危険内容

- 弾幕、ボス、演出、UI、音、世界観、Docker、開発ハーネスなど、面白い要素が多く、広げようと思えばいくらでも広がる
- 初期段階から全部盛りにすると進まなくなる

### 破綻の兆候

- 仕様書ばかり増えて実装が始まらない
- 最小試作ができる前にサブ要素が増える
- 完成条件が毎回遠のく

### 避け方

- 最初の完成条件を小さく固定する
- 初回目標は「1つの綺麗な弾幕パターンが遊べること」に絞る
- 新要素追加は、土台完成後に行う

## 危険ポイント 4: 演出優先で基礎実装が遅れること

### 危険内容

- 見た目の魅力が強い企画なので、背景、ボス立ち絵、UI装飾、スペル演出に先に手を出しやすい
- その結果、移動、弾、判定、ゲームオーバーといった基礎が後回しになる

### 破綻の兆候

- 見た目はあるのにゲームとして成立していない
- プレイフィールの調整に入れない
- 変更のたびに全体が崩れる

### 避け方

- 先にゲームループを完成させる
- 自機移動、弾、敵、被弾を優先する
- 演出は「成立したものを飾る」順番にする

## 危険ポイント 5: 開発環境の複雑化

### 危険内容

- Elixir, Phoenix, Babylon.js, Docker, Windows, Docker Desktop という要素が重なっている
- 環境問題で止まると、ゲーム開発に入る前に消耗しやすい

### 破綻の兆候

- 動作確認の前にセットアップ調整だけで時間が溶ける
- ホスト依存とコンテナ依存が混ざる
- `mix` をどこで実行するか曖昧になる

### 避け方

- Docker を正本に固定する
- `mix` 系コマンドはコンテナ内実行に統一する
- 追加コンテナを増やしすぎない
- まずは `app` 1 コンテナで成立させる

## 危険ポイント 6: Elixirで全部やりたくなること

### 危険内容

- Elixir 主導の思想が強いため、描画や即時反応まで全部 Elixir 側で背負いたくなる可能性がある
- それをやると、設計が重くなり、ブラウザ側の自然な役割まで奪ってしまう

### 破綻の兆候

- 小さな表示変更でもサーバ側変更が必要になる
- 描画都合がゲームロジックに侵食する
- フロントエンドが単なる受け身を超えて不自然に複雑になる

### 避け方

- ゲーム状態は Elixir、描画は Babylon.js と役割を固定する
- 描画専用の都合を Elixir の中核ロジックへ混ぜない
- 変換層を薄く保つ

## 危険ポイント 7: Codex向け文書が増えすぎること

### 危険内容

- 文書化は強いが、多すぎると読む順番や責務が曖昧になる
- 1つの変更で複数文書が矛盾し始める可能性がある

### 破綻の兆候

- どの文書が正なのかわからない
- 似たことが複数の場所に書かれている
- 更新漏れが増える

### 避け方

- 文書ごとの責務を守る
- `AGENTS.md` で読む順番を固定する
- 変更時は対応する文書だけを更新する
- 新文書は必要になってから増やす

## 最初の成功条件

最初の成功条件の正本は [FIRST_MILESTONE.md](FIRST_MILESTONE.md) とする。
このプロジェクトが破綻せず進んでいると判断できる最初の基準は次の状態である。

- Docker 上で Phoenix が起動する
- 自機が動く
- 敵が1体出る
- 敵弾が1種類出る

ここまで到達できれば、企画は現実的な進行段階に入ったとみなせる。
被弾判定や美しい弾幕パターンは、その次の段階で確認する。

## 進め方の原則

- 迷ったら小さい方を選ぶ
- 迷ったら視認性を優先する
- 迷ったら演出より操作感を優先する
- 迷ったら Docker と Elixir の責務分離を守る

## 結論

このプロジェクトの難しさは高い。
ただし、難所は明確であり、順番を守れば十分に挑戦可能である。
危険なのは企画そのものではなく、要素を同時に広げすぎる進め方である。
